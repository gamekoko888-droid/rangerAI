# 代码审查 Skill

## 审查流程

### 第一步：快速扫描（30 秒）

拿到代码后，先回答三个问题：
1. 这段代码的**目的**是什么？
2. 代码量大约多少行？复杂度如何？
3. 使用了什么语言/框架？

### 第二步：四维审查

#### 维度一：安全性（Critical）

| 检查项 | 严重度 | 示例 |
|--------|--------|------|
| SQL 注入 | 🔴 Critical | 字符串拼接 SQL 查询 |
| XSS | 🔴 Critical | 未转义的用户输入渲染到 HTML |
| 硬编码密钥 | 🔴 Critical | API key、密码写在代码里 |
| 路径遍历 | 🟠 High | 用户输入直接拼接文件路径 |
| 不安全的反序列化 | 🟠 High | pickle.loads(user_input) |
| CORS 配置过宽 | 🟡 Medium | Access-Control-Allow-Origin: * |
| 缺少输入验证 | 🟡 Medium | 未验证参数类型/范围 |

#### 维度二：性能（High）

| 检查项 | 影响 | 示例 |
|--------|------|------|
| N+1 查询 | 数据库压力 | 循环中逐条查询 |
| 内存泄漏 | 服务崩溃 | 未关闭的连接/文件句柄 |
| 无缓存 | 响应慢 | 重复计算相同结果 |
| 阻塞操作 | 并发差 | 同步 I/O 在异步上下文 |
| 无分页 | 内存溢出 | 一次加载全部数据 |

#### 维度三：可维护性（Medium）

| 检查项 | 影响 | 示例 |
|--------|------|------|
| 函数过长 | 难理解 | 单函数超过 50 行 |
| 命名不清 | 难理解 | `x`, `tmp`, `data2` |
| 重复代码 | 难维护 | 相同逻辑出现 3+ 次 |
| 缺少注释 | 难接手 | 复杂算法无说明 |
| 深层嵌套 | 难阅读 | if-if-if-if 4 层以上 |
| 魔法数字 | 难理解 | `if status == 3` |

#### 维度四：最佳实践（Low）

| 检查项 | 语言 | 建议 |
|--------|------|------|
| 错误处理 | 通用 | 不要 catch 所有异常后忽略 |
| 类型安全 | TS/Python | 使用类型注解 |
| 测试覆盖 | 通用 | 关键路径需要单元测试 |
| 日志规范 | 通用 | 用结构化日志替代 print |
| 依赖管理 | 通用 | 锁定版本号 |

### 第三步：输出审查报告

```markdown
# 代码审查报告

## 概述
- 文件：{文件名}
- 语言：{语言}
- 行数：{行数}
- 总体评级：{A/B/C/D/F}

## 🔴 Critical Issues（必须修复）
### Issue 1: {标题}
- 位置：第 {行号} 行
- 问题：{描述}
- 风险：{可能的后果}
- 修复建议：
```{语言}
// 修复前
{原代码}
// 修复后
{修复代码}
```

## 🟠 Important Issues（建议修复）
...

## 🟡 Minor Issues（可选优化）
...

## ✅ 做得好的地方
- {值得肯定的设计/实现}

## 总结
{1-3 句话概括代码质量和改进方向}
```

### 评级标准

| 等级 | 标准 | 含义 |
|------|------|------|
| A | 0 Critical, 0-1 Important | 生产就绪 |
| B | 0 Critical, 2-3 Important | 小修后可上线 |
| C | 0 Critical, 4+ Important | 需要较多改进 |
| D | 1+ Critical | 有安全/稳定性风险 |
| F | 3+ Critical | 需要重写 |

## 语言特定检查

### Python
- 使用 `with` 管理资源
- 避免 mutable default arguments
- 使用 f-string 而非 % 或 .format()
- 使用 pathlib 而非 os.path
- 类型注解（Python 3.9+）

### JavaScript/TypeScript
- 使用 `const`/`let` 而非 `var`
- 避免 `==`，使用 `===`
- Promise 错误处理（.catch 或 try-catch）
- 避免回调地狱
- TypeScript 严格模式

### Go
- 错误处理不要忽略（`_ = err`）
- 使用 context 传递超时
- goroutine 泄漏检查
- defer 的执行顺序

## 审查礼仪

- 先肯定做得好的地方，再提改进建议
- 区分"必须修复"和"建议优化"
- 提供具体的修复代码，不只是指出问题
- 解释"为什么"而不只是"怎么改"
