**R37 任务书**

**版本定位**
R37 不追求“再补一个零散能力”，而是正面解决 R36 暴露的三条主短板：

1. **代码执行能力弱**：当前只有 shell，无真正沙箱/可控执行闭环  
2. **部署稳定性不够**：caddy / file-server inactive，线上链路不稳  
3. **浏览器能力未走主路径**：browser 有能力，但 Gateway 仍偏向 web_fetch

**R37 核心定调**
**从“能力存在”升级到“能力真正走通主链路”。**

R36 的问题不是没功能，而是：
- 统计修复多，真实能力增量少
- 有 browser / TTS / planner，但不稳定、不默认、不高频
- 可观测性有了，但还不能支撑“真实压测验收”

所以 R37 的目标不是堆新模块，而是把 **执行、稳定性、主路径调用率** 三件事打透。

---

**R37 总目标**

目标评分：**5.2 → 6.8~7.2**

必须拿下的提升：

- **代码执行**：2.0 → 5.5+
- **部署稳定性**：4.5 → 6.0+
- **浏览器自动化**：4.0 → 5.5+
- **任务完成率可信度**：从“记录改善”变成“真实压测可验证”

**范围边界**

**本轮纳入：**
- 沙箱/受控代码执行
- caddy / file-server 稳定性修复
- Gateway 工具选择优化
- 真实压测与验收数据沉淀
- Gap Analysis R37 更新

**本轮不纳入：**
- 大规模前端新功能
- 全新多代理架构重写
- 长链路自治重构
- 大规模数据库 schema 改造

原则：**只做能直接抬分、能被验收、能进入主链路的事。**

**T1 受控代码执行环境 v1**

**目标**
补齐 Manus 差距里最大的短板：**可隔离的执行环境**。

**为什么做**
R36 最大失分项就是代码执行 2.0。没有沙箱，就没有：
- 安全执行用户代码
- 复杂脚本试跑
- 可复现的自动修复闭环
- “文件即上下文”的大任务处理能力

**实现要求**
1. 提供受控执行入口
   - 独立工作目录
   - 超时控制
   - stdout/stderr 截断
   - 退出码回传
2. 支持基础语言运行
   - Node.js
   - Python
   - Bash
3. 支持最小依赖安装策略
   - 白名单/临时目录安装
   - 自动清理
4. 支持文件产物回收
   - 生成文件列表
   - 大输出落盘，不直接灌上下文
5. 事件埋点
   - code_exec_started
   - code_exec_finished
   - code_exec_failed
   - artifact_written

**DoD**
- 3 类标准任务通过：
  1. 运行 hello-world 脚本
  2. 读取输入文件并生成输出文件
  3. 安装轻量依赖并运行
- 每类任务至少测 5 次，总成功率 ≥80%
- 超时/异常有结构化错误返回
- 输出大于阈值时自动写文件而非塞满上下文

**风险**
- 依赖安装污染宿主环境
- 长输出撑爆上下文
- 超时/僵尸进程清理不干净

**控制**
- 临时目录执行
- 超时 kill + 清理
- 输出上限 + 文件化回传

---

**T2 caddy + file-server 稳定性修复**

**目标**
解决 R36 明确指出的部署稳定性短板。

**实现要求**
1. 检查服务状态、启动失败原因、端口占用
2. 修复启动配置/依赖问题
3. 建立最小健康检查
   - 进程 active
   - 本地端口监听
   - HTTP 200 / 302
4. 纳入统一巡检脚本
5. 输出故障归因
   - 配置错误
   - 端口冲突
   - 权限问题
   - 路径不存在
   - 上游依赖异常

**DoD**
- caddy active
- file-server active
- 连续 3 次健康检查通过
- 重启后状态稳定
- 关键 URL 可返回预期状态码

**验收证据**
- systemctl status / service status
- 端口监听结果
- curl 返回
- 健康检查日志

**T3 Gateway 工具选择优化**

**目标**
让 browser 在应使用时进入默认路径，而非被 web_fetch 抢走。

**问题本质**
R36 的 browser_action=1，本质是 prefetch 绕过去了，不是 Executor 真选了 browser。

**实现要求**
1. 梳理任务类型路由规则
   - page_lookup
   - site_navigation
   - interactive extraction
   - auth-required page
   - structured static fetch
2. 明确 browser 优先条件
   - 需要点击/输入/滚动
   - 页面依赖 JS 渲染
   - 目标内容需交互后可见
   - 需要截图/元素定位
3. 明确 web_fetch 优先条件
   - 纯静态内容抓取
   - 快速摘要
   - 文本型页面直接提取
4. 加埋点
   - tool_route_candidate
   - tool_route_chosen
   - tool_route_fallback
5. 支持失败回退
   - browser → web_fetch
   - web_fetch → browser（当静态抓取不足）

**DoD**
- 50 条 web 类压测任务
- browser 在应触发场景中的选用率 ≥40%
- 路由理由可审计
- fallback 有记录，不是静默失败
