# RangerAI 模型治理规则
**版本**: v4.0 (迭代质量治理)
**生效日期**: 2026-04-29
**更新**: v4.0 — 基于 R71-R86 迭代质量审查，新增规则九至规则十四，系统性解决死代码、验收形式化、修 bug 引入 bug、用户价值缺失、迭代质量不稳定五大问题。

---

## 规则一：R70 主从协作架构
**核心原则：GPT-5.5 是大脑（规划+验收），V4 Pro 是双手（执行所有代码操作）。**

所有文件（无例外）统一执行流程：
GPT-5.5 规划（生成 taskBrief + acceptanceCriteria）→ V4 Pro 执行 → GPT-5.5 验收（关键步骤）

V4 Pro 可以修改任何文件，包括核心模块文件。质量由 GPT-5.5 的验收环节保障，而非限制执行权限。

---

## 规则二：GPT-5.5 验收策略
关键写入步骤（代码变更、配置修改、部署）→ reviewPolicy=gpt_review
只读/检查步骤 → reviewPolicy=auto_pass
最后一步 → 强制 gpt_review

验收不通过时：GPT-5.5 注入修正建议，V4 Pro 重试一次。仍不通过则强制通过并记录。

---

## 规则三：熔断机制
1. 同一个 bug 修了 2 次仍未解决 → GPT-5.5 接管根因分析
2. 修复引入了新 bug → GPT-5.5 评估是否需要回滚
3. 需要同时修改 4 个以上文件 → GPT-5.5 重新拆分任务粒度

## 规则四：部署前强制检查清单
1. .mjs 语法检查: node --check
2. 前端构建成功: npx vite build 退出码 0
3. 服务端口 LISTENING
4. HTTP 200: curl https://ranger.voyage/

## 规则五：模型分工矩阵（R70 统一架构）
所有任务类型统一：GPT-5.5 规划 + V4 Pro 执行 + GPT-5.5 验收（关键步骤）

## 规则八：代码重构验收强化（v3.1）

**背景**：R81 迭代中 V4Pro 将 `const ctxMgr = getContextManager(...)` 移入 `try` 块内部，但 `try` 块外的 `return new Promise(...)` 中有 5 处引用 `ctxMgr`。由于 `const/let` 的块级作用域，`try` 块结束后 `ctxMgr` 不可访问，导致 100% 必现的 `ReferenceError: ctxMgr is not defined`。GPT-5.5 验收时仅检查了"任务目标是否完成"，未做作用域分析，bug 逃逸。

**强制规则**：凡涉及以下操作的步骤，GPT-5.5 验收时 **必须** 执行对应检查：

### 8.1 变量声明移动
当一个变量的 `const/let/var` 声明位置被移动时：
1. 用 `grep -n '变量名'` 列出该变量的 **所有引用行号**
2. 确认新声明位置的作用域 **覆盖所有引用点**（特别注意 `try/catch/if/for/while/switch` 等块级边界）
3. 如果引用点跨越了声明所在的 `{}` 块 → **必须将声明提升到最近的公共父作用域**

### 8.2 代码块结构变更
当 `try-catch`、`if-else`、循环等块结构被添加、删除或移动时：
1. 检查块内所有 `const/let` 声明的变量是否在块外被引用
2. 如果是 → 将声明移到块外，块内改为赋值

### 8.3 强制语法验证
任何涉及 `.mjs` 文件结构变更的步骤，验收时必须执行：
```bash
node -c <被修改的文件>
```
语法检查不通过 → 验收直接失败，不允许 auto_pass。

### 8.4 作用域速查表（V4Pro 执行参考）
| 声明方式 | 作用域 | 跨 try/catch | 跨 if/else | 跨函数 |
|---------|--------|-------------|-----------|--------|
| `const` | 块级 `{}` | ❌ 不可 | ❌ 不可 | ❌ 不可 |
| `let` | 块级 `{}` | ❌ 不可 | ❌ 不可 | ❌ 不可 |
| `var` | 函数级 | ✅ 可以 | ✅ 可以 | ❌ 不可 |

**安全做法**：需要跨块引用的变量，在块外用 `let` 声明并初始化为 `null`，块内仅做赋值。

## 规则九：禁止写了不用（Dead Code Prevention）（v4.0 新增）

**背景**：R71-R86 审查发现 4 个模块共 740 行代码（task-workspace、circuit-breaker、quality-scorer、worker-result-schema）写完后从未被生产代码调用。R75 需求文档自己指出"5 个 0 引用模块"的问题，然后 R75 自己又新增了死代码。

**强制规则**：

### 9.1 当轮接入原则
新写的模块/函数 **必须在当轮迭代中被至少一个生产代码路径调用**。如果当轮无法接入调用链，**不要写这个模块**。

### 9.2 GPT-5.5 规划阶段门控
GPT-5.5 在生成 taskBrief 时，对每个新模块必须回答：
- **谁调用它？**（具体文件名和行号范围）
- **本轮能接入吗？**（如果不能，移到下一轮需求文档，本轮不写）

如果回答不了"谁调用它"→ 该模块 **禁止在本轮创建**。

### 9.3 验收阶段引用检查
GPT-5.5 验收新模块时，**必须执行**：
```bash
grep -rn '模块名' /opt/rangerai-agent/worker/*.mjs | grep -v '模块名.mjs' | grep -v '//'
```
返回 0 行（无生产引用）→ **验收失败**，要求 V4Pro 补充调用方代码或删除该模块。

### 9.4 季度清理
每 20 轮迭代后，执行一次死代码扫描：
```bash
for f in worker/*.mjs; do
  name=$(basename "$f" .mjs)
  refs=$(grep -rn "$name" worker/*.mjs | grep -v "$f" | grep -v '//' | wc -l)
  [ "$refs" -eq 0 ] && echo "DEAD: $f"
done
```
DEAD 模块 → 移到 `archive/dead-code-$(date +%Y%m%d)/`。

## 规则十：端到端验收（Runtime Verification）（v4.0 新增）

**背景**：R74 验收标记 P1-2 "✅ 已实现"，但 R80 发现恢复动作根本没执行。验收只做了 `node --check` 和导出验证，从未做过运行时验证。

**强制规则**：

### 10.1 三级验收体系
每个 feat/fix 提交的验收 **必须包含以下三级**，缺一不可：

| 级别 | 内容 | 方法 | 失败处理 |
|------|------|------|---------|
| L1 语法 | 代码能解析 | `node --check` | 直接失败 |
| L2 单元 | 函数能调用、返回正确 | 导出验证 / smoke test | 直接失败 |
| L3 运行时 | 功能在真实环境中工作 | 发送测试消息触发完整执行链路 | 标记为"未验证"，禁止标记 ✅ |

### 10.2 L3 运行时验证方法
对于 worker 核心链路的改动（openclaw-handler、planner、task-engine、sub-agent-orchestrator），验收时必须：
1. 重启 rangerai-ws（通过 defer-restart）
2. 通过 ranger.voyage 发送一条简单测试消息（如"你好"）
3. 检查 worker 日志中无新的 Error/ReferenceError/TypeError
4. 确认消息得到正常回复

如果 L3 验证失败 → 立即回滚到上一个 git tag。

### 10.3 验收报告格式升级
验收报告中每个子项的状态标记规则：
- ✅ = L1 + L2 + L3 全部通过
- ⚠️ = L1 + L2 通过，L3 未执行或未通过
- ❌ = L1 或 L2 未通过

**禁止在 L3 未通过的情况下标记 ✅。**

## 规则十一：修复副作用检查（Fix Side-Effect Gate）（v4.0 新增）

**背景**：R81 修复 ctxMgr TDZ 时引入了更严重的作用域 bug。R76 修复 R74 遗漏的 `index` undefined。修 bug 引入新 bug 是反复出现的模式。

**强制规则**：

### 11.1 修复影响面扫描
每个 fix 提交前，V4Pro **必须执行**：
```bash
# 列出被修改变量/函数的所有引用点
grep -n '被修改的标识符' <文件> | head -20
```
GPT-5.5 验收时检查：修改是否影响了列出的引用点？如果影响了 → 逐一确认每个引用点仍然正确。

### 11.2 错误路径覆盖
修复涉及 try-catch、retry、fallback 等错误处理路径时，验收 **必须确认**：
- 正常路径（happy path）仍然工作
- 错误路径（error path）按预期触发
- retry 路径的变量在 retry 作用域内可访问

### 11.3 回归检测
修复提交后，如果 worker 日志在 5 分钟内出现新的 Error 类型（之前不存在的）→ **自动触发回滚**：
```bash
git stash && sudo systemctl restart rangerai-ws
```

## 规则十二：用户价值节奏（User Value Cadence）（v4.0 新增）

**背景**：R71-R86 共 16 轮迭代，零个直接面向用户的功能改进。全部是内部架构重构。

**强制规则**：

### 12.1 五轮一交付
每连续 5 轮内部迭代后，**下一轮必须包含至少 1 个用户可感知的改进**。用户可感知的改进包括但不限于：
- 响应速度提升（可量化）
- 新工具/能力支持
- 错误提示改善
- 前端 UI 优化
- 新的业务功能

### 12.2 需求文档必填字段
每轮迭代的需求文档（r{N}-requirements.md）**必须包含**：
```
## 用户价值
本轮迭代对用户的直接影响：<具体描述 / "无直接影响（内部重构）">
距上次用户可感知改进：<N 轮>
```
当"距上次用户可感知改进"≥ 5 时，GPT-5.5 **必须**在需求中加入至少一个用户可感知的改进项。

### 12.3 迭代类型标签
每次 git commit 必须在 message 中标注迭代类型：
- `[USER]` — 用户可感知的改进
- `[INFRA]` — 内部架构/基础设施
- `[FIX]` — bug 修复
- `[HYGIENE]` — 清理/测试/文档

## 规则十三：迭代质量门控（Quality Gate）（v4.0 新增）

**背景**：25 小时 41 次提交，平均 37 分钟一次。前一轮的 bug 在后一轮才发现，修复链条越来越长。

**强制规则**：

### 13.1 单轮完整性要求
每轮迭代 **必须** 在同一轮内完成完整闭环：
```
写代码 → L1 语法验证 → L2 单元验证 → L3 运行时验证 → 验收报告
```
**禁止**"本轮写代码，下轮再测试"或"本轮写模块，下轮再接入"。

### 13.2 修复链长度限制
如果一个功能的修复链超过 3 轮（例如 R74 写 → R76 修 → R80 又修），GPT-5.5 必须：
1. 停止继续打补丁
2. 回滚到功能引入前的状态
3. 重新设计方案，一次性实现

### 13.3 提交频率限制
单轮迭代的提交数 **不超过 5 个**。如果需要更多提交，说明任务拆分粒度不够，GPT-5.5 应重新拆分。

### 13.4 迭代间冷却
每轮迭代完成后，**必须等待 L3 运行时验证通过**才能开始下一轮。禁止在上一轮 L3 未通过的情况下开始新的 feat 开发。

## 规则十四：自我迭代审计日志（Iteration Audit Trail）（v4.0 新增）

**背景**：R71-R86 期间缺乏系统性的迭代质量追踪，问题只在外部审查时才被发现。

**强制规则**：

### 14.1 迭代记分卡
每轮迭代完成后，在验收报告末尾追加记分卡：
```
## 迭代记分卡
- 新增代码行数: <N>
- 新增模块数: <N>（其中已接入调用链: <N>）
- 修复的 bug 数: <N>（其中由本系列迭代引入: <N>）
- L3 运行时验证: PASS / FAIL / SKIP
- 用户可感知改进: 有（<描述>）/ 无
- 死代码新增: <N> 行
```

### 14.2 累计健康指标
在 `memory/iteration-health.md` 中维护累计指标：
```
| 指标 | 当前值 | 健康阈值 | 状态 |
|------|--------|---------|------|
| 连续内部迭代轮次 | <N> | ≤5 | 🟢/🔴 |
| 死代码总行数 | <N> | ≤200 | 🟢/🔴 |
| 最近5轮修复链最大长度 | <N> | ≤2 | 🟢/🔴 |
| L3 验证通过率 | <N>% | ≥80% | 🟢/🔴 |
```
任何指标变红 → GPT-5.5 在下一轮规划中 **必须优先处理**。

## 规则十五：连续执行（Continuous Execution）（v4.1 新增）

**背景**：Ranger 在执行多步任务时，每完成一个步骤就停下来等用户确认才继续下一步。Manus 不会这样——Manus 拿到任务后一口气执行完所有步骤，只在需要用户输入时才暂停。

**15.1 连续执行原则**
- 任务一旦开始，**必须连续执行到完成**，步骤间自动过渡
- LLM 的文字输出 ≠ 等待用户回复。进度通知是单向的
- GPT-5.5 在生成计划时，**禁止**在步骤间插入"等待用户确认"节点

**15.2 允许暂停的唯一条件**
- 需要用户提供信息（密码、API Key、选择方案）
- 不可逆操作前的确认（删除生产数据、支付、发布）
- 步骤预算用尽（超过 30 步）

**15.3 进度汇报方式**
- 通过 `[NOTIFY]` 标记输出，每 5-8 个工具调用一句（≤50 字）
- 汇报完立即继续下一步，**不等待用户回复**
- 最终结果在所有步骤完成后一次性交付

**GPT-5.5 验收**：如果 Ranger 在步骤间停下等用户确认（非上述允许条件），验收失败。

## 规则十六：高风险操作路由（High-Risk Operation Routing）（v4.1 新增）

**背景**：R94 密钥迁移（修改 systemd service 文件）被路由到 V4 Pro 执行。高风险操作应由 GPT-5.5 执行或审查。

**16.1 高风险操作定义**
以下操作属于高风险，**必须路由到 GPT-5.5**：
- 修改 systemd service/timer 文件
- 修改 nginx/caddy 配置
- 修改 .env 文件或密钥/凭证
- 执行 `rm -rf`、`DROP TABLE`、`DELETE FROM`、`TRUNCATE`
- 修改 iptables/ufw 防火墙规则
- 修改 SSL/TLS 证书配置
- 部署到生产环境
- 数据库 schema 迁移（ALTER TABLE）

**16.2 代码级实现**
`smart-router.mjs` 的 `smartRouteByPhase()` 已添加 `HIGH_RISK_KEYWORDS` 检测：
- 当步骤内容包含高风险关键词时，自动升级到 `openai/gpt-5.5`
- 即使 phase 是 `coding` 或 `sysadmin`，也会被拦截升级
- 日志标记：`[GOVERNANCE] High-risk operation detected`

**16.3 GPT-5.5 规划时的责任**
- 在生成计划时，如果步骤涉及高风险操作，**必须**将 phase 标记为 `review` 或 `validation`
- 不得将高风险操作标记为 `coding` 或 `sysadmin`

**GPT-5.5 验收**：如果高风险操作被 V4 Pro 执行（日志中无 GOVERNANCE 升级标记），验收失败。
